# Bifrost

SQL Server data migration tool for .NET 10. Named after the rainbow bridge connecting realms in Norse mythology.

## Projects

- **Bifrost.Core** — shared migration logic (models, database, export, import, direct, restructure, retry, history)
- **Bifrost.CLI** — command line interface
- **Bifrost.GUI** — Avalonia cross-platform desktop app
- **Bifrost.Tests** — xUnit unit tests

## Build

```bash
dotnet build
dotnet test
```

## Releases

Releases are built automatically via GitHub Actions when a version tag is pushed:

```bash
git tag v1.0.0
git push origin v1.0.0
```

This runs all tests, publishes self-contained single-file executables for Windows x64 and Linux x64, and attaches them as zips to a GitHub Release. Tags containing a hyphen (e.g. `v1.0.0-beta`) are marked as pre-releases.

To build manually:

```bash
dotnet publish Bifrost.CLI -r win-x64 -c Release --self-contained -p:PublishSingleFile=true
dotnet publish Bifrost.GUI -r win-x64 -c Release --self-contained -p:PublishSingleFile=true

dotnet publish Bifrost.CLI -r linux-x64 -c Release --self-contained -p:PublishSingleFile=true
dotnet publish Bifrost.GUI -r linux-x64 -c Release --self-contained -p:PublishSingleFile=true
```

## CLI Usage

### export
Exports tables from the source database to `.sql` files. Use when source and target are on different networks.

```bash
bifrost export
bifrost export --config config.json --output ./output
bifrost export --config config.json --dry-run
```

### import
Imports `.sql` files generated by `export` into the target database. Each file is wrapped in a transaction — a failure rolls back that file cleanly.

```bash
bifrost import
bifrost import --config config.json --output ./output
bifrost import --config config.json --dry-run
```

### direct
Migrates directly from source to target with no intermediate files. Use when both servers are reachable on the same network.

```bash
bifrost direct
bifrost direct --config config.json
bifrost direct --config config.json --bulk
bifrost direct --config config.json --dry-run
bifrost direct --config config.json --bulk --dry-run
```

The `--bulk` flag uses `SqlBulkCopy` for significantly faster transfers.

The `--dry-run` flag connects to the source, counts rows, and logs what would happen without writing anything to the target.

### restructure
Moves tenant tables into named schemas and strips the tenant ID suffix from table names. Detects naming conflicts before starting and rolls back completed moves if a failure occurs mid-run.

```bash
bifrost restructure
bifrost restructure --config config.json
```

### status
Connects to both source and target and reports table counts per database, plus the last 5 run history entries.

```bash
bifrost status
bifrost status --config config.json
```

## Config file format

All commands share a single config file.

```json
{
  "source": {
    "server": "192.168.1.100",
    "port": 1433,
    "username": "sa",
    "password": "yourpassword",
    "encrypt": false,
    "trustServerCertificate": true
  },
  "target": {
    "server": "192.168.2.100",
    "port": 1433,
    "username": "sa",
    "password": "yourpassword",
    "encrypt": false,
    "trustServerCertificate": true
  },
  "databases": [
    {
      "comment": "Tenant filter example",
      "sourceDatabase": "MainAppDB",
      "targetDatabase": "MainAppDB_Staging",
      "tableFilter": "tenant",
      "tenantId": "142",
      "overrides": [
        { "name": "dbo.HugeTable_142", "where": "CreatedAt >= '2024-01-01'" },
        { "name": "dbo.BigTable_142", "query": "SELECT TOP 1000 * FROM [dbo].[BigTable_142] ORDER BY CreatedAt DESC" },
        { "name": "dbo.UnwantedTable_142", "ignore": true }
      ]
    },
    {
      "comment": "Explicit table list",
      "sourceDatabase": "ReportingDB",
      "targetDatabase": "ReportingDB_Staging",
      "tableFilter": "explicit",
      "tables": [
        "dbo.SmallTable",
        { "name": "dbo.HugeTable", "where": "CreatedAt >= '2024-01-01'" },
        { "name": "dbo.UnwantedTable", "ignore": true }
      ]
    },
    {
      "comment": "All tables",
      "sourceDatabase": "LookupDB",
      "targetDatabase": "LookupDB_Staging",
      "tableFilter": "all",
      "overrides": [
        { "name": "dbo.LargeAuditLog", "ignore": true }
      ]
    }
  ],
  "tenants": [
    {
      "comment": "Move and rename tenant tables",
      "tenantId": "142",
      "schema": "WarehouseCo",
      "database": "MainAppDB_Staging",
      "createCompatibilityViews": true
    },
    {
      "comment": "Move all tables from an existing schema without renaming",
      "tenantId": "",
      "schema": "WarehouseCo",
      "database": "MainAppDB_Staging",
      "sourceSchema": "test",
      "createCompatibilityViews": false
    }
  ]
}
```

## tableFilter options

| Value | Description |
|-------|-------------|
| `tenant` | All tables ending with `_tenantId` or `tenantId` |
| `explicit` | Only the listed tables |
| `all` | All tables in the database |

## Per-table override options

| Field | Description |
|-------|-------------|
| `ignore: true` | Skip this table |
| `where: "..."` | Filter rows with a WHERE clause |
| `query: "..."` | Use a fully custom SELECT query |

## Tenant restructure options

| Field | Description |
|-------|-------------|
| `tenantId` | Suffix to match and strip from table names (e.g. `142`) |
| `schema` | Target schema to move tables into |
| `database` | Database to restructure (uses target connection) |
| `sourceSchema` | If set, moves all tables from this schema as-is without suffix matching or renaming |
| `createCompatibilityViews` | Creates views in the original schema pointing to the new location |

## Reliability

- **Retry** — failed tables are retried up to 3 times with a 2 second delay before being counted as failures
- **Transactions** — each imported `.sql` file runs inside a transaction and is rolled back cleanly on failure
- **Restructure rollback** — if a table fails mid-restructure, all completed moves for that tenant are reversed
- **Conflict detection** — restructure scans for naming collisions before starting and aborts if any are found

## GUI

The GUI stores named configs in `bifrost-configs.json` next to the executable. Run history is stored in `bifrost-history.json`.

- **Import config** — load a `.json` config file and save it as a named config (`↑` button)
- **Export config** — save a named config to a standalone `.json` file (`↓` button per config)
- **Dry Run** — checkbox to preview an operation without writing anything
- **SqlBulkCopy** — checkbox to enable bulk mode for direct migration
- **Progress bar** — shows per-table progress during any operation
- **Run history** — last 20 runs shown in the sidebar with outcome, operation, config name, and duration
